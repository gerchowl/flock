title: Persistence and remote access
description: Detach from Flock, reattach later, use named sessions, and connect over SSH.
---

Flock keeps panes running in a background server. Your terminal client can detach and reconnect later.

For the local, SSH, and `flock --remote` workflows, see [How to work with Flock](/docs/how-to-work/).

## Detach and reattach

Start Flock:

```bash
flock
```

Detach the client with `ctrl+b q`.

Your panes and agents keep running. Reattach by running Flock again:

```bash
flock
```

If you want to end the session and stop its panes, stop the default server:

```bash
flock server stop
```

When Flock starts again after a full server stop, it restores the saved session shape. For what survives detach, server restart, screen history replay, native agent session restore, and live handoff, see [Session state and restore](/docs/session-state/).

## Named sessions

Use named sessions when you want independent Flock servers.

```bash
flock session list
flock session attach work
flock session attach side-project
flock session stop work
flock session delete side-project
```

A named session has its own panes, tabs, workspaces, sockets, and runtime state. It still shares the same global config file.

Use `--json` for scripts:

```bash
flock session list --json
flock session stop work --json
flock session delete side-project --json
```

## Remote attach over SSH

Run Flock on a server and attach from any SSH client:

```bash
ssh you@server
flock
```

In this mode your shell is remote, Flock runs on the remote host, and panes keep running there after you detach.

You can also attach through SSH from your local machine:

```bash
flock --remote workbox
flock --remote ssh://you@server:2222
```

In this mode your local Flock is a thin client. It connects over SSH, starts or attaches to the remote Flock server, and streams the UI back to your local terminal. Because the client runs locally, Flock can bridge local desktop features such as image clipboard paste into the remote session by copying the image to a remote temp file and pasting that path.

By default, `flock --remote` uses your local Flock keybindings for that attach. This keeps local muscle memory even when the remote server has different config. The local keybindings are a snapshot from attach time; detach and reattach after editing local keybindings. Use `--remote-keybindings server` when you want the remote server config instead. Local custom command keybindings are not sent, because those commands would run on the remote host.

For repeat targets, use your SSH config:

```text
Host workbox
  HostName server.example.com
  User you
  Port 2222
```

Then attach with:

```bash
flock --remote workbox
```

Remote attach supports Linux and macOS hosts on x86_64 and aarch64. Flock checks the remote platform, prefers a matching `flock` already on the remote `PATH`, then checks `~/.local/bin/flock`. If no matching binary exists, interactive runs prompt to install one to `~/.local/bin/flock`; non-interactive runs fail instead of modifying the host. If `~/.local/bin` is not on the remote `PATH`, Flock warns after install.

By default, `flock --remote` runs the bridge through a temporary SSH config that includes your SSH config first, then adds fallback keepalive settings. Existing user keepalive settings win. Set `[remote].manage_ssh_config = false` to use plain `ssh` without Flock's generated bridge config.

By default, remote attach uses the normal restart/stop flow if it needs to replace or restart a running remote server. To opt into experimental live handoff for a supported running remote server, pass `--handoff`:

```bash
flock --remote workbox --handoff
```

If you SSH into the server first and run `flock` there, Flock runs entirely on the server. That mode is useful and simple, but it cannot access your local desktop clipboard beyond normal terminal text paste.

When your local and remote platforms match, Flock can copy the current local binary for direct installs. For Homebrew, mise, and Nix installs, or when the platforms differ, it downloads the matching release asset for the current client version from `https://flock.dev/latest.json`.

For local builds or custom binaries, set `FLOCK_REMOTE_BINARY` to a local file path before running remote attach.

```bash
FLOCK_REMOTE_BINARY=target/release/flock flock --remote workbox
```

## Remote named sessions

Use `--session` with `--remote` to attach to a named session on the remote host:

```bash
flock --remote workbox --session agents
```

## Direct terminal attach

Full Flock attach opens the whole workspace UI. Direct attach opens one server-owned terminal in your current terminal.

Attach by agent target:

```bash
flock agent attach reviewer
```

Attach by terminal ID:

```bash
flock terminal attach term_abc123
```

Direct attach streams the current rendered terminal state, then live ANSI frames. Input goes straight to that terminal.

Detach with `ctrl+b q`. Send a literal `ctrl+b` with `ctrl+b ctrl+b`.

Only one writable direct attach client owns input and resize for a terminal. Use `--takeover` to replace an existing owner:

```bash
flock terminal attach term_abc123 --takeover
```

## Single-process escape hatch

Use `--no-session` to run Flock without the background server/client split:

```bash
flock --no-session
```

This is mainly an escape hatch for debugging or compatibility. The default persistent session mode is the normal path.
